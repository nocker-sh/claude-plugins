# 記述ルール

## 基本方針

- コードから分かることは書かない
- 判断基準: 「コードをgrep/readすれば分かるか?」→ Yes なら書かない
- コードから読み取れない意図や背景だけ残す

## 書くもの

- なぜその判断をしたか
- 却下した選択肢とその理由
- 顧客の生の声
- 製品の方向性と境界
- 計算式・仕組み（金額の算定式・按分・KPI の定義・集計の考え方）。コードを読めば追えるが、顧客が結果を信頼するために業務の言葉で残す価値がある。コード識別子は使わず、式と意味で書く

## 書かないもの

- APIエンドポイント一覧
- 関数、クラス、コンポーネントの一覧
- データ構造の詳細（テーブル名・カラム名など内部識別子）
- ファイルパス（src/... など）
- 実装手順
- UIデザイン・レイアウト。画面部品の名前（カード・バッジ・タブ・パネル・進捗バー・フォーム・モーダル・ダイアログ・グリッド・サイドバーなど）、配置・色・クリック等の操作記述。「どう見えるか」ではなく「何のための機能か・どんな価値か」を書く

## メタ記述の禁止

.docs/ の各ファイルは製品の内容だけで構成する。ドキュメント自身を説明する文章を書かない。

- 読み方の案内を書かない。「本ドキュメントの読み方」「まず〜を読んでから〜を読む」「対象読者は〜」のような導入
- 書き方・更新手順を書かない。「このファイルには〜の形式で記載する」「新しい項目は〜に追記する」「テンプレートに従って書く」のような運用説明
- 書式・構成・運用ルールの正本はこのスキルの references/docs/ であり、.docs/ 側に複製しない。書き方を残したくなったらスキル側を更新する

## リンクはwikiリンク

Obsidianで閲覧する前提。Markdownリンク `[text](path)` は使わない。

- ファイルへのリンク: `[[slug]]`
- 表示名を変える場合: `[[slug|表示名]]`
- パス(`./path/to/file.md`)は禁止。ファイル名だけで解決する

## 図はmermaid

flowchart, sequenceDiagram, erDiagramを使用。
