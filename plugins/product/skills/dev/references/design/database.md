# Database Design

## Naming

- テーブル名・カラム名に製品固有の世界観・呼称を持ち込まない。汎用名を使う
- 例: 利用者の呼称が「メンバー」「ゲスト」「クルー」などでも、テーブルは `users`
- UI 文言とスキーマを分離する。世界観は表示層で被せる
- 理由: 製品名・コンセプトは変わるがデータ構造は残る。リネーム移行コストを避ける
- 複数形にできないリソース名は使わない（`info`、`metadata`、`data` など）。粒度が曖昧な証拠なので分解して具体的な複数形名にする

## Deletion

- 物理削除が正。論理削除（`deleted_at` / `is_deleted`）は負債
- 理由: すべてのクエリに削除フィルタが必要になり、ユニーク制約・外部キー・集計が壊れる
- 例外: 監査要件・復元要件がある場合のみ、選択肢として採用可。採用時は負債と認識する
- 履歴が必要なら別テーブル（`*_archives` など）に移送する
